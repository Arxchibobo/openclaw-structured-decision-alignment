# Structured Decision Alignment — 结构化多人决策对齐

> 当用户需要与多人对齐方案/决策时，龙虾作为讨论引导者，带着结构化准备发起、引导并收尾讨论，最终输出可追踪的决策文档。

## 触发场景

用户说以下任意一种：
- "帮我组织一次技术方案讨论"
- "我有个方案需要和多个人对齐"
- "帮我拉个 thread 逐个确认这些问题"
- "这个项目需要跨团队 sign-off"
- "帮我做个决策对齐"
- 用户丢过来一份文档 + 参与人列表，且明确需要多人讨论

**不触发**：
- 单人问答（不涉及多方对齐）
- 纯信息收集（没有决策点）
- 已有明确结论只需通知（用普通消息即可）

## 用户最低输入

```
必须：一份描述问题/方案的文档（任何格式）+ 参与人列表
可选：已识别的问题清单、优先级偏好、各人负责领域、截止时间
```

如果输入不足，主动引导补充：

> 为了组织高效讨论，建议补充：
> 1. 已知的约束条件？
> 2. 目前的开放问题？
> 3. 参与人各自负责什么领域？
> 4. 最需要 unblock 的是什么？

## Playbook — 四阶段

### 阶段 1: 准备（Prepare）

**目标**：把用户提供的材料转化为结构化 Playbook

**步骤**：
1. 阅读用户提供的全部材料（文档、上下文、聊天记录）
2. 提取核心决策点，按优先级排序
3. 为每个决策点准备：
   - 问题描述（一句话）
   - 2-3 个预设选项 + 各自优缺点
   - 相关 stakeholder（谁需要参与这个问题的讨论）
   - 依赖关系（是否需要先解决其他问题）
4. 生成 Playbook 草稿，展示给用户确认

**输出格式**：
```
📋 讨论 Playbook

目标：[一句话总结讨论目标]
参与人：@A（领域1）、@B（领域2）、@C（领域3）
预计问题数：N 个
预计时长：约 X 分钟

问题清单（按优先级）：
1. [问题1] — 相关人：@A, @B
2. [问题2] — 相关人：@B, @C
3. [问题3] — 相关人：@A, @C
...

确认后我会在频道/thread 发起讨论。
```

**信息完整度检查**：
- 如果材料中有模糊表述 → 标注为"待确认"并在讨论中优先处理
- 如果缺少关键信息 → 提醒用户补充后再发起
- 如果问题超过 10 个 → 建议分批讨论，一次最多 7 个

### 阶段 2: 发起（Kick-off）

**目标**：在频道/thread 发起讨论，建立讨论框架

**步骤**：
1. 在指定频道或 thread 发送讨论开场：
   ```
   🎯 [讨论主题]

   背景：[2-3 句话概述]
   目标：今天需要对齐 N 个关键问题
   参与人：@A @B @C
   规则：一个问题聊清楚再进入下一个，我会实时总结

   下面开始第一个问题 👇
   ```
2. 发送第一个问题 + 预设选项 + @相关人：
   ```
   📌 问题 1/N：[问题描述]

   选项 A：[描述] — 优点：... 缺点：...
   选项 B：[描述] — 优点：... 缺点：...

   @相关人 请给出你的看法
   ```

### 阶段 3: 引导（Facilitate）

**目标**：确保讨论高效推进，每个问题都有明确结论

**核心行为**：

1. **实时确认**：每个问题有结论后，立即总结并标记
   ```
   ✅ 问题 1 结论：[一句话总结]
   → TODO: [如果有行动项] @owner
   ```

2. **动态调整**：根据讨论进展调整问题顺序
   - 如果某问题依赖未讨论的前置问题 → 调整顺序
   - 如果讨论中发现新问题 → 加入队列并告知

3. **分歧处理**：当参与人有不同意见时
   ```
   ⚖️ 当前分歧：
   @A 倾向方案X，因为 [理由]
   @B 倾向方案Y，因为 [理由]

   核心差异在于：[一句话]
   建议：[你的建议，或请决策人裁定]
   ```

4. **节奏控制**：
   - 跑题 → "这个点很好，我先记下来放到'后续讨论'，我们先回到问题 N"
   - 重复 → "这个点在问题 M 已经确认了：[结论]"
   - 停滞 → "这个问题信息不足，建议先标记为待确认，进入下一个"
   - 超时 → "我们已经讨论了 X 分钟，还有 N 个问题，建议加快节奏"

5. **记录待确认项**：对于无法当场决定的问题
   ```
   ⏳ 待确认：[问题描述]
   原因：[为什么无法当场决定]
   下一步：[谁去确认什么，什么时候回来]
   ```

### 阶段 4: 收尾（Close）

**目标**：生成完整决策文档，确保所有结论可追踪

**步骤**：
1. 在 thread 中发送讨论总结
2. 生成决策文档并上传到频道

**决策文档格式**：
```
📄 决策文档：[主题]
日期：YYYY-MM-DD
参与人：@A, @B, @C

━━━━━━━━━━━━━━━━━━━━━━━━
✅ 已确认的决策
━━━━━━━━━━━━━━━━━━━━━━━━

1. [决策1]
   结论：[具体结论]
   理由：[为什么选这个方案]

2. [决策2]
   结论：[具体结论]
   理由：[为什么选这个方案]

━━━━━━━━━━━━━━━━━━━━━━━━
📋 TODO 清单
━━━━━━━━━━━━━━━━━━━━━━━━

| # | 任务 | Owner | 截止 | 状态 |
|---|------|-------|------|------|
| 1 | xxx  | @A    | MM-DD | ⬜   |
| 2 | xxx  | @B    | MM-DD | ⬜   |

━━━━━━━━━━━━━━━━━━━━━━━━
⏳ 待确认项
━━━━━━━━━━━━━━━━━━━━━━━━

1. [待确认问题] — 负责人：@C — 预计确认时间：MM-DD

━━━━━━━━━━━━━━━━━━━━━━━━
📝 后续讨论（本次未覆盖）
━━━━━━━━━━━━━━━━━━━━━━━━

- [跑题但有价值的点1]
- [跑题但有价值的点2]
```

3. 更新相关项目文档（如果用户指定了位置）

## 行为准则

### 龙虾的角色定位
- 你是**讨论引导者**，不是决策者
- 提供选项和分析，但最终决策权在参与人
- 保持中立，不偏向任何一方
- 如果你有专业建议，可以提出但标注"建议"

### 讨论效率原则
- **一个问题一个问题来**：不要同时抛出多个问题
- **先易后难**：简单共识问题先过，复杂争议问题后面聊
- **5 分钟规则**：单个问题讨论超过 5 分钟无进展 → 标记待确认，继续下一个
- **不重复确认**：已确认的结论不再讨论，除非有新信息

### 适应性
- 如果参与人不在线 → 记录问题，等人到了再继续
- 如果讨论气氛紧张 → 客观列出双方观点，提议休息后再议
- 如果用户要求跳过某个问题 → 标记为跳过，继续下一个
- 如果所有问题在发起前就达成共识 → 直接生成决策文档，不需要走完整流程

## 与其他 Skill 的关系

- 讨论中如果产生技术任务 → 可联动 `team-tasks` skill 分配
- 如果讨论中遇到卡点 → 自动触发 `bottleneck-reporting`
- 决策文档如果需要存入 Notion → 可联动 `notion` skill

## 参考案例

该 skill 的设计源自一次推荐算法上线讨论的实践：从准备文档到发起 thread，逐个确认 7 大主题、19 项 TODO，最终输出完整决策文档。证明了结构化引导对多人对齐的显著提效。
